# Phase 1.6a — Daemon Module 架構 + Session 重設計

> 日期：2026-03-22
> 狀態：Draft
> 關聯：Issue #46

## 概述

將 daemon 從單體 server 重構為 Module 架構，並重新設計 Session 管理：以 tmux 為 SOT、DB 降級為 meta cache、移除主動同步。本次只實作 Module 骨架和 Session module，Stream / CC-detect 遷移留給 1.6b。

## 設計決策

### Daemon 定位：Thin Relay

- 被動回應 SPA 請求，不主動管理業務邏輯
- 每台 daemon 只管自己主機上的資源
- Tab / Workspace / Session menu 順序等 UI 狀態不存 daemon
- 多 Host 場景下每台 daemon 獨立運作，跨 host 協調由 SPA 處理

### tmux 為 SOT

- Session 列表永遠即時查 tmux，不從 DB 讀取
- DB 只存無法從 tmux 即時取得的 meta（mode、cc_session_id、cc_model、cwd）
- 能從 tmux 即時取得的資訊（name、pane_current_command 等）不存 meta
- tmux session 消失 → 對應 meta 在下次存取時清掉（無主動同步、無輪詢、無 hook）

## 一、Module 架構（A+ 方案）

### Core

Daemon 核心，持有共用基礎設施，不含業務邏輯。

```go
type Core struct {
    cfg      *config.Config
    tmux     tmux.Executor
    meta     *store.MetaStore
    events   *EventsBroadcaster
    registry *ServiceRegistry
    modules  []Module
}
```

| 元件 | 職責 |
|------|------|
| `config.Config` | TOML 配置 |
| `tmux.Executor` | tmux CLI 抽象（已有 Real/Fake 實作） |
| `store.MetaStore` | SQLite meta cache（精簡後） |
| `EventsBroadcaster` | WebSocket 事件廣播 |
| `ServiceRegistry` | Module 間 Interface Registry |

### Module Interface

```go
type Module interface {
    Name() string
    Init(core *Core) error
    RegisterRoutes(mux *http.ServeMux)
    Start(ctx context.Context) error
    Stop() error
}
```

- `Init`：取得共用資源、向 Registry 註冊自己的能力
- `RegisterRoutes`：掛載 HTTP / WebSocket 路由
- `Start`：啟動背景工作（如有）
- `Stop`：優雅關閉

### Service Registry

```go
type ServiceRegistry struct {
    services map[string]any
}

func (r *ServiceRegistry) Register(name string, svc any)
func (r *ServiceRegistry) Get(name string) (any, bool)
```

Module 在 `Init` 階段註冊能力（如 `session.provider`），其他 module 在 `RegisterRoutes` 或 `Start` 階段透過 interface 查詢取用。Module 之間不直接 import。

Event broadcaster 用於廣播通知（fire-and-forget），Service Registry 用於 request-response 互動。

### 啟動順序

```
1. 載入 config
2. 建立 Core（tmux executor, meta store, events, registry）
3. 依 config [features] 決定啟用哪些 module
4. 全部 module.Init()（註冊能力到 registry）
5. 全部 module.RegisterRoutes()
6. 全部 module.Start()
7. 啟動 HTTP server
```

所有 `Init` 完成後才進入 `RegisterRoutes` 和 `Start`，保證 registry 查詢時依賴已就位。

### Module 劃分

| Module | 類型 | Phase |
|--------|------|-------|
| **session** | 核心（必載） | 1.6a |
| **stream** | 核心（必載） | 1.6b |
| **cc-detect** | 核心（必載） | 1.6b |
| **fs** | 擴充（config 啟用） | 未來 |
| **git** | 擴充（config 啟用） | 未來 |

兩類 module：
- **Session 衍生**（stream、cc-detect）：依賴 session module 的 `SessionProvider` interface
- **獨立資源**（fs、git）：不依賴 session，基底不是 terminal

### Config 擴充

```toml
[features]
fs = false
git = false
# 核心 module（session, stream, cc-detect）永遠啟用，不列在此
```

## 二、Session Module

### 職責

- tmux session 列表（即時查詢 tmux）
- Session meta CRUD（DB cache）
- PTY WebSocket relay（terminal 檢視）
- Session ID 編碼（6 碼 code ↔ tmux ID）

### SessionProvider Interface

註冊到 Service Registry 供其他 module 取用：

```go
type SessionProvider interface {
    ListSessions() ([]SessionInfo, error)
    GetSession(code string) (*SessionInfo, error)
    HandleTerminalWS(w http.ResponseWriter, r *http.Request, code string)
}
```

`HandleTerminalWS` 包辦完整 PTY relay 生命週期（WebSocket 升級、tmux attach、sizing、雙向轉發），呼叫方不需要碰 PTY handle。

### SessionInfo

```go
type SessionInfo struct {
    // 即時從 tmux 取得（不存 DB）
    Code             string   // 6 碼 base36（API / URL 用）
    TmuxID           string   // "$0"（內部用）
    Name             string   // tmux session name
    Exists           bool
    CurrentCommand   string   // pane_current_command

    // Meta cache（存 DB）
    Mode             string   // "term" | "stream" | "jsonl"
    CCSessionID      string
    CCModel          string
    Cwd              string
}
```

### DB Schema

```sql
-- 取代現有 sessions table
CREATE TABLE session_meta (
    tmux_id       TEXT PRIMARY KEY,  -- "$0", "$1"...
    mode          TEXT DEFAULT 'term',
    cc_session_id TEXT DEFAULT '',
    cc_model      TEXT DEFAULT '',
    cwd           TEXT DEFAULT '',
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 移除
-- DROP TABLE groups;
-- sessions table 的 sort_order, group_id, uid, name 欄位移除
```

主鍵為 tmux session ID。注意事項：
- tmux session ID 在同一 tmux server 生命週期內不可變（rename 不影響）
- **tmux server 重啟後 ID 會重新分配**，舊 meta 變成孤立紀錄。這是可接受的——孤立 meta 在下次 `GET /api/sessions` 時會被清理（tmux 不存在 → 刪 meta），meta 本身只是 cache，丟失無害
- `name` 不存 DB（可從 tmux 即時取得），API 回傳的 name 來自 tmux 即時查詢

### Session ID 編碼

tmux session ID（`$N`）和 6 碼 base36 code 之間的雙向轉換。

```go
func EncodeSessionID(tmuxID string) string   // "$0" → "k7r2fm"
func DecodeSessionID(code string) string     // "k7r2fm" → "$0"
```

- XOR + bit shuffle 實作，確保連續 ID 產生無規律的 code
- XOR key 和 bit shuffle permutation 為 compile-time 常數（所有 daemon 實例一致）
- Parse 規則：`$N` → 取出整數 N（`strconv.Atoi(tmuxID[1:])`）
- 31 bit 空間落在 6 碼 base36 以內（36^6 > 2^31）；N 超出 31 bit 時回傳錯誤（實務不會發生）
- 轉換只在 API 邊界發生
- SPA / URL 全程使用 code，不暴露 tmux ID

### API 路由

| 方法 | 路由 | 行為 | 歸屬 |
|------|------|------|------|
| GET | `/api/sessions` | 即時 `tmux list-sessions` + 合併 DB meta。tmux 不存在的 meta 在此清除 | session module |
| POST | `/api/sessions` | `tmux new-session` + 寫入 meta | session module |
| DELETE | `/api/sessions/{code}` | `tmux kill-session` + 清除 meta | session module |
| WS | `/ws/terminal/{code}` | PTY relay。session 不存在 → 回報錯誤 + 清 meta | session module |
| POST | `/api/sessions/{code}/mode` | 切換 mode（更新 meta） | session module |
| GET | `/api/sessions/{code}/history` | CC JSONL 歷史 | 暫留 server.go（1.6b 遷移） |
| POST | `/api/sessions/{code}/handoff` | Handoff 協調 | 暫留 server.go（1.6b 遷移） |
| WS | `/ws/cli-bridge/{session}` | Relay producer | 暫留 server.go（1.6b 遷移） |
| WS | `/ws/cli-bridge-sub/{session}` | SPA subscriber | 暫留 server.go（1.6b 遷移） |
| WS | `/ws/session-events` | 事件廣播 | Core 層（非 module） |

1.6b 遷移前，暫留 server.go 的 handler 透過 `SessionProvider` interface 存取 session 資訊（使用 code 而非舊 ID）。

### Meta 清理策略

不額外做定期清理。清理發生在自然的存取時機：

- `GET /api/sessions`：查到 tmux 不存在 → 清對應 meta
- `WS /ws/terminal/{code}`：連線時 session 不存在 → 回報錯誤 + 清 meta
- 其他 API 存取到不存在的 session 時同理

### 移除項目

- `syncTmuxSessions()` 函式
- `groups` table
- `sessions` table 的 `sort_order`、`group_id`、`uid` 欄位
- Session 自增 ID（改用 tmux_id 為主鍵）

## 四、遷移策略

### DB 遷移

- 新增 `session_meta` table
- 從現有 `sessions` table 遷移 mode、cc_session_id、cc_model、cwd
- 需要一次性查詢 tmux 取得各 session 的 tmux_id，透過 name 對應到現有紀錄
- **tmux 不可用時的 fallback**：跳過舊 meta 遷移，直接建立空的 `session_meta` table（meta 是 cache，丟失無害）
- 遷移完成後移除舊 `sessions` 和 `groups` table

### 程式碼遷移

1. 建立 `internal/core/` package（Core + ServiceRegistry + Module interface）
2. 建立 `internal/module/session/` package（Session module）
3. 重寫 `cmd/tbox/main.go` 使用 Core 啟動流程
4. 現有 session_handler.go 邏輯遷入 session module
5. 現有 terminal_handler.go 邏輯遷入 session module
6. 修改 `tmux.Executor.ListSessions()` 的 format string，增加 `#{session_id}` 欄位；`TmuxSession` struct 新增 `ID string` field
7. `StartStatusPoller` 暫留 server.go，改為透過 `SessionProvider.ListSessions()` 取得 session 列表（不再直接呼叫 store）
8. Stream / CC-detect / Handoff 暫時留在 server.go（1.6b 遷移），透過 SessionProvider interface 存取 session

### SPA 影響

- Session API response 格式變更（id → code，移除 sort_order / group_id）
- WebSocket URL 使用 code（`/ws/terminal/{code}`）
- Tab store 的 session 識別改用 code
- 移除 SPA 端對 group_id 的依賴（如有）
