# Phase 1.6b — Stream + CC Module 遷移

> 日期：2026-03-23
> 狀態：Draft
> 前置：Phase 1.6a（PR #59, #60）
> 關聯：#61（handleCreate 並發）、#62（Terminal 渲染）

## 概述

將 legacy `internal/server/` 中剩餘的 handler 遷移為獨立 module，完成 daemon 的完全 module 化。遷移後 legacy server 目錄整個移除，legacy store（`state.db`）一併淘汰。

主要工作：
1. Core 層擴充（EventsBroadcaster、Config endpoint、Stop 改造、依賴排序）
2. 新增 CC module（detect + history + CC 操作原子）
3. 新增 Stream module（bridge handler + handoff 編排）
4. 全面統一 session code（Bridge WS URL、Events 廣播、SPA store key）
5. TDD 方式遷移測試，刪除 legacy server + legacy store

---

## 一、Core 層擴充

### 1.1 EventsBroadcaster 移入 Core

**現況**：`server/events_handler.go`（173 LOC），WebSocket pub-sub，廣播 session 狀態 / relay / handoff / init 事件。

**變更**：移至 `internal/core/events.go`，由 Core 持有 `core.Events`。所有 module 透過 `Init(core *Core)` 時存取 `core.Events.Broadcast()`。

**Broadcast session key 統一用 code**：`Broadcast(sessionCode, eventType, value)`。所有呼叫端改傳 session code 而非 name。SPA 端對應的 store key 也從 name 改為 code（見第四節）。

**On-subscribe callback**：EventsBroadcaster 提供 `OnSubscribe(func(conn *websocket.Conn))` 機制。新 WS 連線建立時，依序呼叫所有已註冊的 callback，讓各 module 推送自己的 snapshot：
- cc module 註冊：推送 status snapshot（各 session 的 CC 偵測狀態）
- stream module 註冊：推送 relay snapshot（各 session 的 relay 連線狀態）

這避免了 Core 直接依賴 module 的 service。

**理由**：EventsBroadcaster 是跨 module 共用基礎設施，與 config、tmux executor 同級。

### 1.2 Config HTTP Endpoint 移入 Core

**現況**：`server/config_handler.go`（105 LOC），`GET/PUT /api/config`。

**變更**：移至 Core 層，Core 在 `RegisterRoutes()` 時註冊 config endpoint。Config 變更時透過 `core.Events` 廣播通知。

**理由**：Config 是全域基礎設施，只有 105 LOC，不值得獨立 module。

### 1.3 Module Stop 改造

**現況**：`Stop()` 不帶 context，且 `core.Stop()` 在 `cancel()` 後才呼叫各 module 的 `Stop()`。Module 無法做 graceful shutdown timeout 控制。

**變更**：

```go
// Module interface 變更
Stop(ctx context.Context) error  // 取代 Stop()

// Core.Stop() 新實作
func (c *Core) Stop() error {
    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    var errs []error
    for i := len(c.modules) - 1; i >= 0; i-- {
        if err := c.modules[i].Stop(ctx); err != nil {
            errs = append(errs, err)
        }
    }
    c.cancel()  // 最後才取消 core context
    return errors.Join(errs...)
}
```

關鍵：
- 新建 shutdown context（帶全域 10s timeout），不是用 core ctx
- `cancel()` 移到最後，module Stop 期間 core ctx 仍有效
- 收集所有 errors，不 fail-fast，每個 module 都有機會清理

**注意**：此為 interface breaking change，已實作的 session module 需同步修改 `Stop()` 簽章為 `Stop(ctx context.Context) error`。

### 1.4 宣告式依賴排序

**現況**：Module 初始化順序靠 `main.go` 中 `AddModule()` 的呼叫順序。

**變更**：Module interface 新增 `Dependencies() []string`，Core 在 Init 前用 Kahn's algorithm 拓撲排序。偵測到循環依賴回 error。

```go
// Module interface 新增
Dependencies() []string

// 各 module 宣告
func (m *SessionModule) Dependencies() []string { return nil }
func (m *CCModule) Dependencies() []string      { return []string{"session"} }
func (m *StreamModule) Dependencies() []string   { return []string{"session", "cc"} }
```

**理由**：`[features]` config 允許使用者 enable/disable optional module（fs / git），手動排序在動態啟停場景不可行。~35 行拓撲排序換掉隱式順序依賴。

## 二、CC Module（`module/cc/`）

### 定位

輕量的 Claude Code 相關業務集合。與 CC 有直接關聯的小範圍功能統一歸納於此。Stream 因體量大且獨立，不包含在內。

### 結構

| 檔案 | 職責 |
|------|------|
| `module.go` | Module 生命週期 + route 註冊 |
| `handler.go` | HTTP handler（history endpoint） |
| `detect.go` | 包裝 `internal/detect/`，實作 `CCDetector` interface |
| `operator.go` | CC 操作原子（Exit / Launch / Interrupt / GetStatus） |
| `history.go` | 包裝 `internal/history/`，實作 `CCHistoryProvider` interface |

### Registry 暴露（三個 interface 分開註冊）

```go
// CCDetector — 偵測 CC 狀態
// 回傳 Status 不帶 error，與底層 detect.Detector.Detect() 一致
type CCDetector interface {
    Detect(tmuxTarget string) Status
}

// CCHistoryProvider — CC 對話歷史
// 參數為 cwd（用於建構 JSONL 路徑），非 tmuxTarget
type CCHistoryProvider interface {
    GetHistory(cwd string, ccSessionID string) ([]map[string]any, error)
}

// CCOperator — CC 操作原子
// Launch 接受完整命令字串（由 orchestrator 組裝），非 args slice
type CCOperator interface {
    Exit(ctx context.Context, tmuxTarget string) error
    Launch(ctx context.Context, tmuxTarget string, cmd string) error
    Interrupt(ctx context.Context, tmuxTarget string) error
    GetStatus(ctx context.Context, tmuxTarget string) (*StatusInfo, error)
}
```

**參數型別說明**：`tmuxTarget` 接受 `session_name:0` 格式（指定 window 0）。這是 send-keys、capture-pane 等操作需要的精確 pane 定位。呼叫端從 `SessionInfo` 取得 session name 後自行組裝 `name:0`。注意 terminal attach（PTY WS）使用不帶 `:0` 的 bare name，兩者是不同場景（見 commit `eb8ce35`）。

**Registry key 常數**：`"cc.detector"`、`"cc.history"`、`"cc.operator"`（對應 session module 的 `"session.provider"` pattern）。

**依賴**：`SessionProvider`（從 Registry 取得，用於 code → tmux ID 轉換）

### Route

| Method | Path | 說明 |
|--------|------|------|
| GET | `/api/sessions/{code}/history` | CC 對話歷史 |

### Status Poller

目前 status poller 在 `server.go` 的 `StartStatusPoller()` 中，定期呼叫 detector 並透過 EventsBroadcaster 廣播。遷移後由 cc module 的 `Start()` 啟動 poller goroutine，透過 `core.Events` 廣播。

Poller 使用 `SessionProvider.ListSessions()` 取得 session 列表（已含 code），對每個 session 組裝 `name:0` 呼叫 `Detect()`，變更時透過 `core.Events.Broadcast(sessionCode, "status", ...)` 廣播。

cc module 在 `Start()` 時透過 `core.Events.OnSubscribe()` 註冊 status snapshot callback，新 WS 連線時推送各 session 的當前偵測狀態。

## 三、Stream Module（`module/stream/`）

### 定位

管理 stream/jsonl 模式的資料串流管道：relay 連線管理、SPA subscriber 扇出、Terminal ↔ Stream 模式切換編排。

### 結構

| 檔案 | 職責 | 估計 LOC |
|------|------|----------|
| `module.go` | Module 生命週期 + route 註冊 + Bridge 實例持有 | ~50 |
| `handler.go` | HTTP 入口 + per-session lock + WS upgrade + init metadata 捕獲 | ~150 |
| `orchestrator.go` | Handoff 編排 + mode revert + 輔助函式 | ~300 |

### Bridge 實例所有權

Bridge（`internal/bridge/`）由 stream module 在 `Init()` 時建立並持有。Bridge 只有 stream 模式在用，不屬於 Core 基礎設施。

stream module 在 `Start()` 時透過 `core.Events.OnSubscribe()` 註冊 relay snapshot callback，新 WS 連線時推送各 session 的當前 relay 連線狀態。

### Init metadata 捕獲

`handleCliBridge`（relay 端 WS）中有 one-shot init metadata 捕獲邏輯：從 relay 的 stream JSON 中擷取 CC model 並更新。遷移後歸入 stream module 的 `handler.go`，透過 `SessionProvider.UpdateMeta()` 更新 MetaStore，不再寫入 legacy store。

### Mode revert

`revertModeOnRelayDisconnect`（relay 斷線時自動將 mode 回復為 term）歸入 stream module 的 `orchestrator.go`。遷移時移除 legacy store 的 fallback 路徑，統一使用 `SessionProvider.UpdateMeta()`。

### 依賴（全從 Registry 取得）

| Interface | 用途 |
|-----------|------|
| `SessionProvider` | 查 session 資訊、code → tmux ID 轉換、更新 mode |
| `CCOperator` | Handoff 時操作 CC（exit / launch / interrupt） |
| `CCDetector` | Handoff 前偵測 CC 狀態 |

### Route

| Method | Path | 說明 |
|--------|------|------|
| POST | `/api/sessions/{code}/handoff` | 觸發模式切換 |
| WS | `/ws/cli-bridge/{code}` | Relay 端連線 |
| WS | `/ws/cli-bridge-sub/{code}` | SPA subscriber 連線 |

### 底層依賴（不動）

- `internal/relay/`（150 LOC）— subprocess ↔ daemon WS，精簡且職責單一
- `internal/bridge/`（140 LOC）— relay ↔ subscriber pub-sub，同上

## 四、Session Code 全面統一

### Bridge WS URL

```
/ws/cli-bridge/{session}      →  /ws/cli-bridge/{code}
/ws/cli-bridge-sub/{session}  →  /ws/cli-bridge-sub/{code}
```

- Bridge 內部（`internal/bridge/bridge.go`）的 map key 也改用 code
- 轉換邊界在 stream module 的 handler 層，透過 `SessionProvider.GetSession(code)` 完成
- Relay 子程序由 daemon 啟動，daemon 組裝連線 URL 時直接用 code

### Events 廣播

`EventsBroadcaster.Broadcast()` 的 session 參數統一使用 code：
- Status poller：`Broadcast(sessionCode, "status", ...)`
- Handoff：`Broadcast(sessionCode, "handoff", ...)`
- Bridge relay 連線/斷線：`Broadcast(sessionCode, "relay", ...)`
- Init metadata：`Broadcast(sessionCode, "init", ...)`

### SPA 端同步修改

SPA 的 session event 處理流程以 `event.session` 為 key 操作 streamStore。改用 code 後需同步修改：

| 檔案 | 修改 |
|------|------|
| `spa/src/lib/session-events.ts` | `SessionEvent.session` 欄位語義從 name → code |
| `spa/src/hooks/useSessionEventWs.ts` | 事件處理以 code 為 key；`.find()` 比對從 `s.name` 改為 `s.code` |
| `spa/src/hooks/useRelayWsManager.ts` | bridge-sub WS URL 改用 code |
| `spa/src/stores/useStreamStore.ts` | `relayStatus`、`sessionStatuses`、`handoffProgress` 等 map 的 key 從 name → code |

SPA 已在 1.6a 全面持有 session code，這些修改是機械性的 key 替換。

### Codec 存取原則

各 module 不直接使用 codec（code ↔ tmux ID 編解碼），一律透過 `SessionProvider.GetSession(code)` 取得完整 session 資訊。Codec 維持為 session module 的內部實作。

## 五、Legacy 清理

### 策略：TDD 遷移

1. 根據新 module 的 interface 寫新 test（定義預期行為）
2. 實作 module 讓新 test 通過（參考舊 handler 邏輯但不受舊結構限制）
3. 新 test 全過後，刪除舊 test + 整個 `internal/server/` 目錄

**理由**：直接搬舊 test 會把舊的耦合（依賴 `server.store`、`server.handleHandoff` 等）帶進新架構，限制改版後的寫法。

### 測試處置

| 舊測試檔 | LOC | 處置 |
|----------|-----|------|
| `session_handler_test.go` | 156 | 直接刪除（session module 已有） |
| `legacy_test.go` | 58 | 直接刪除 |
| `server_test.go` | 129 | 直接刪除（lifecycle 改由 Core 管） |
| `bridge_handler_test.go` | 234 | 新 test in stream module |
| `handoff_handler_test.go` | 524 | 新 test in stream module |
| `e2e_test.go` | 205 | 新 test in stream module |
| `events_handler_test.go` | 343 | 新 test in Core |
| `config_handler_test.go` | 372 | 新 test in Core |
| `history_handler_test.go` | 127 | 新 test in cc module |
| `middleware_test.go` | 112 | 搬至 `internal/middleware/` |

### Legacy Store 完全移除

遷移後所有功能改用 `SessionProvider` + `MetaStore`，legacy store 的使用場景全部消失。1.6b 完成後完全移除：

- `internal/store/store.go`（legacy store 實作）
- `state.db` 檔案
- `main.go` 中的 `store.Open()` 和 `MigrateFromLegacy()` 呼叫

MetaStore（`meta.db`、`internal/store/meta.go`）保留，是 1.6a 設計的正式元件。

開發階段，破壞式變更無需向下相容。

### Middleware 搬遷

`internal/server/middleware.go`（86 LOC）搬至 `internal/middleware/`。Middleware 不是 module 也不是 Core 的職責，是 HTTP server 層的橫切關注點，`main.go` 直接引用。

### 刪除清單

- `internal/server/` 整個目錄
- `internal/store/store.go`（legacy store）
- `main.go` 中的 `store.Open()`、`store.MigrateFromLegacy()` 呼叫

## 六、Module 初始化順序

依賴關係（由 `Dependencies()` 宣告）：

```
session（無依賴）
  ↑
  cc（依賴 session）
  ↑
  stream（依賴 session, cc）
```

Core 在 `Init()` 前自動拓撲排序，`main.go` 中 `AddModule()` 的順序不影響結果。
